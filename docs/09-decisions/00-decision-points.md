# Decision Points 후보 정리 (Working Draft)

> **상태**: Draft — DP 선정 논의용 작업 문서
> **목적**: 7장(Solution Strategy)·9장(ADR) 작성에 앞서, 본 과제에서 **가장 중요하면서 어려운 결정 지점(Decision Point, DP)** 4개를 선정하기 위한 후보 풀을 정리한다.
> **DP 정의**: 특정 QA 문제를 해결하기 위해 서로 트레이드오프가 발생하는 후보안 2~3개를 비교·평가하여 선택하는 아키텍처 결정 지점. 선정된 DP는 각각 ADR로 문서화되며 10장(ATAM)의 Sensitivity/Tradeoff Point 분석 입력이 된다.

---

## 0. 분류 체계

아키텍트가 과제 수행 중 실제로 부딪힌 고민을 세 층위로 분류한다.

| 층위 | 성격 | 처리 방침 |
|------|------|-----------|
| **BD** (Background Decision) | 과제 성립 여부를 가르는 전제 결정. 이미 방향이 정해졌거나 DP로 다루기에는 층위가 다름 | 7장 솔루션 전략의 **배경 설계** 절에 서술 (DP 아님) |
| **DP 후보** | 후보안이 구조적으로 갈리고, 선택을 되돌리기 어려우며, 복수 QA에 상반된 영향을 주는 결정 | 비교·평가 후 DP 확정 → ADR 작성 |
| **TQ** (Tactic Question) | 단일 QA를 개선하는 설계 전술의 탐색 질문. 전술 풀 안에서 구조적 갈림길이 발견되면 DP로 승격 | 전술 후보 풀 정리 → DP 승격 여부 판단 |

## 1. BD-1 — 웹 에이전트 vs "agent-browser" 도구

> 원 고민 #1: 웹 태스크를 **웹 에이전트**로 만들어 argot의 서브 에이전트로 붙이느냐, vs 웹 에이전트까지는 필요 없고 **agent-browser라는 도구(tool)** 로서 웹 태스크를 처리하느냐.

- **성격**: 본 과제(자체 에이전트 루프를 갖는 AI Web Agent)의 **성립 전제**에 해당하는 결정. "에이전트가 필요 없다" 쪽을 택하면 본 ADS 자체가 무의미해지므로, 트레이드오프 비교로 다루는 DP가 아니라 **이미 내려진 배경 결정의 논거를 기록**하는 대상이다.
- **구도**:
  - **A. 웹 에이전트** — 자체 에이전트 루프(계획·도구 호출·관찰·복구)를 가진 서브 에이전트. 다단계 웹 태스크를 자율 수행.
  - **B. agent-browser 도구** — argot(Main Agent)의 루프가 직접 사용하는 브라우저 제어 도구 모음. 루프는 argot 하나만 존재.
- **A를 택한 핵심 논거(기록 대상)**:
  - 웹 태스크는 다단계·장시간·오류복구가 필수인 도메인 — argot 루프에서 도구 호출로 풀면 argot의 컨텍스트가 DOM·페이지 상태로 오염되고 토큰 비용이 Main Agent에 전가됨
  - 브라우저 도메인 지식(DOM 해석, 사이트 패턴, Gen UI)의 캡슐화·독립 배포(NFR-035)
  - 오류 격리 — 웹 태스크 실패가 argot·Bixby 경험을 오염시키지 않아야 함(NFR-009)
- **문서화 위치**: 7장 도입부 "솔루션 전략 배경" 절 (또는 1장 배경과 상호 참조). **DP 목록에서 제외.**

## 2. DP 후보

### DP-C1 — argot 통합 인터페이스: Skill(SKILL.md) vs Sub-Agent Interface

> 원 고민 #2: 웹 에이전트가 **SKILL.md로 붙느냐**, argot의 **서브 에이전트 인터페이스로 붙느냐**.

- **문제**: BD-1에서 웹 에이전트로 가기로 했을 때, argot(Platform Main Agent)에 어떤 계약(contract)으로 통합되는가. 이 결정은 라우팅 정확도·결합도·배포 독립성·컨텍스트/토큰 경계·제어권 이양 방식에 모두 영향을 주는 구조적 결정이다.
- **후보안(초안)**:
  - **A. Skill 방식(SKILL.md)** — 선언적 명세(트리거·능력·사용법)를 argot이 읽고 스킬로 라우팅. argot 관점에서 본 시스템은 "잘 문서화된 도구/스킬".
  - **B. Sub-Agent 인터페이스** — argot의 서브 에이전트 계약(위임·보고·인터럽트 프로토콜)에 따라 독립 에이전트로 등록. 태스크 위임 후 자율 수행·결과 보고.
  - *(C. 하이브리드 — 진입은 Skill 선언으로, 실행은 서브 에이전트 위임으로. argot 코드 확인 후 실재 여부 판단)*
- **예상 평가 축**: Skill 라우팅 정확도(NFR-004) / 모듈 결합도(NFR-024) / 독립 배포(NFR-035) / 오류 격리(NFR-009, TO-4) / 제어권 이양·인터럽트 전파(NFR-034) / 토큰·컨텍스트 경계(NFR-008)
- **미결·선행 작업**: **argot 코드를 직접 확인**하여 두 인터페이스의 실체(계약 형태, 인터럽트/보고 프로토콜, 컨텍스트 공유 범위)를 비교해야 함. 장단점이 극명하게 갈리는지 여부 자체가 미확인 — 갈리지 않으면 DP가 아니라 관례 채택으로 격하될 수 있음.
- **관련 기존 문서**: 6.3.1 (NFR-004 라우팅), 4장 TC(argot 종속 제약), 1장 Out-of-Scope(argot 자체)

### DP-C2 — Quick Action 실행 구조

> 원 고민 #3: 현재는 SKILL.md에 Quick Action을 명시해서 **argot이 Quick Action과 일반 태스크를 분리 실행**하는데, 이 설계가 최선인가? 다른 솔루션은 없는가?

- **문제**: 단순 명령을 에이전트 루프 없이 1초 이내(NFR-001)에 처리하는 우회 경로를 **어느 계층이 소유·분기하는가**. 분기 지점의 위치는 응답성·라우팅 정확도·카탈로그 유지보수·argot 결합도를 동시에 결정한다.
- **후보안(초안)**:
  - **A. argot-레벨 분기(현행)** — SKILL.md에 Quick Action 카탈로그를 선언, argot이 발화 해석 시점에 Quick Action / 일반 태스크를 분기해 별도 호출.
    - 장점 예상: 웹 에이전트 기동 없이 최단 경로, argot의 NLU 재사용
    - 단점 예상: 카탈로그가 argot 배포에 묶임(SKILL.md 갱신 문제), 분기 오류 시 본 시스템이 교정 불가, argot 결합도 증가
  - **B. 웹 에이전트 내부 라우터** — argot은 웹 태스크를 통째로 위임, 본 시스템 진입 후 경량 라우터가 Quick Action / 에이전트 루프를 분기.
    - 장점 예상: 카탈로그 소유권·배포 독립성 확보, 분기 로직을 자체 개선 가능
    - 단점 예상: 진입 오버헤드(기동·IPC)가 1초 예산을 잠식, 라우터 자체의 상주 비용(NFR-006)
  - **C. 2단계 하이브리드** — argot은 coarse 분기(웹 태스크 여부)만, 본 시스템 라우터가 fine 분기(Quick Action 여부). 선언은 최소 힌트만 SKILL.md에.
- **예상 평가 축**: Quick Action ≤1초(NFR-001) / 라우팅 정확도(NFR-004) / 토큰 회피 효과(NFR-008) / 메모리 상주(NFR-006) / 배포 독립성(NFR-035) / 보안 경로 분리(TO-1)
- **의존 관계**: **DP-C1의 결과에 종속** — 통합 인터페이스가 정해져야 분기 지점 후보가 확정된다. 또한 6.8 Open Question 2(Quick Action 후보 명령 집합 카탈로그)와 연동.
- **관련 기존 문서**: 6.6 QAS-001, 6.7 TO-1, 6.8 OQ-2

### DP-C3 — 페이지 컨텍스트 표현 방식 (TQ-2·TQ-4 승격)

- **문제**: LLM에 전달하는 페이지 상태(관찰)를 어떤 표현으로 구성하는가. 토큰 비용의 최대 지배 요인이자 요소 식별 정확도의 상한을 결정하며, 클라우드 전송 범위(프라이버시)와도 직결된다.
- **후보안(초안)**:
  - **A. 경량 텍스트 표현** — 접근성 트리/인터랙티브 요소 목록 등 압축 텍스트만 전송
  - **B. 멀티모달** — 텍스트 표현 + 스크린샷 병용
  - **C. 적응형** — 기본은 A, 요소 식별 실패·모호 시에만 스크린샷 추가 전송
- **예상 평가 축**: 토큰 비용(NFR-008) / 요소 식별 정확도(NFR-021) / 왕복 지연(NFR-030) / 전송 최소화(NFR-018) / 구현 복잡도
- **관련**: TO-2, TO-3, TO-5의 교차점. 상세 전술 풀은 3절 TQ-2·TQ-4 참조.

### DP-C4 — 상주·격리 모델 (TQ-1·TQ-3 승격 + TO-4)

- **문제**: 본 시스템의 컴포넌트를 언제 기동·해제하고(상주 전략), 어떤 입도로 격리하는가(프로세스 vs 인프로세스). 메모리 50MB(NFR-006) ↔ 슬립 복귀 1초(NFR-005) ↔ 호스트 격리 99.95%(NFR-009)의 3자 트레이드오프.
- **후보안(초안)**:
  - **A. 상시 상주 + 강한 격리** — 별도 프로세스 상주. warm-start 최적, 메모리 최악
  - **B. on-demand 기동** — 태스크 시점 기동·유휴 해제. 메모리 최적, 복귀 지연 위험
  - **C. 부분 상주(코어만)** — 경량 라우터·상태만 상주, 루프·도구는 on-demand. 절충안
- **예상 평가 축**: peak 메모리(NFR-006) / 복귀 시간(NFR-005) / 격리 보증(NFR-009) / Quick Action 경로 기동 시간(NFR-001, DP-C2와 연동)
- **관련**: TO-4. 상세 전술 풀은 3절 TQ-1·TQ-3 참조.

### DP-C5 — 브라우저 제어 인터페이스 계층

- **문제**: 에이전트가 브라우저를 **관찰·조작하는 기술 계층**을 무엇으로 하는가. 전 기능이 이 계층 위에 서므로 가장 되돌리기 어려운 결정 중 하나이며, TV 브라우저 엔진에 대한 종속 정도를 결정한다.
- **후보안(초안)**:
  - **A. 원격 디버깅 프로토콜(CDP류)** — 표준적·강력하나 디버깅 포트 상시 개방의 보안 표면, 프로토콜 오버헤드
  - **B. 엔진 내장 자동화/접근성 API** — 오버헤드 최소·통합 깊이 최상이나 엔진 수정·버전 종속
  - **C. in-page 스크립트 주입** — 구현 용이하나 페이지(잠재적 적대 콘텐츠)와 같은 컨텍스트에서 실행되는 보안 위험, CSP 충돌
- **예상 평가 축**: 관찰 충실도·조작 능력(NFR-021) / 오버헤드(NFR-006, NFR-030) / 보안 표면(NFR-012, NFR-014) / 엔진 버전 종속·유지보수(NFR-023, NFR-024) / 모킹·시뮬레이션 용이성(NFR-025)

### DP-C6 — 외부 콘텐츠 신뢰 모델: 인젝션 방어 배치 (TO-1 승격)

- **문제**: 웹 페이지라는 **신뢰 불가 콘텐츠**가 LLM 컨텍스트에 들어가는 구조에서, 프롬프트 인젝션 방어를 파이프라인의 어디에 어떤 방식으로 두는가.
- **후보안(초안)**:
  - **A. 사전 필터링** — 컨텍스트 투입 전 탐지·새니타이즈(분류기/휴리스틱). 빠르나 우회 가능성
  - **B. 권한 분리(이중 LLM)** — 외부 콘텐츠는 비특권 처리 전용으로 격리하고, 행동 결정 컨텍스트에는 원문이 닿지 않게 함. 강하나 토큰·지연 2배 위험
  - **C. 행동 시점 게이팅** — 컨텍스트 오염은 허용하되, 위험 행동(결제·전송·설정 변경)을 정책 엔진 + HITL로 차단. 방어선이 뒤로 이동
- **예상 평가 축**: 인젝션 방어율(NFR-014) / 지연(NFR-030) / 토큰(NFR-008) / 태스크 성공률 부작용(과차단, NFR-021) / DP-C9(HITL)과 연동
- **관련**: TO-1. Quick Action 경로 분리 가설과 결합 가능(경로별 상이한 방어 수준).

### DP-C7 — 행동 공간(Action Space) 추상화 수준

- **문제**: LLM이 선택하는 도구(행동)의 입도를 저수준 프리미티브로 하는가, 고수준 시맨틱 액션으로 하는가. 범용성 ↔ 정확도·비용의 근본 트레이드오프.
- **후보안(초안)**:
  - **A. 저수준 프리미티브** — click/type/scroll + 셀렉터. 임의 사이트에 범용이나 스텝 수·토큰 증가, 실수 표면 큼
  - **B. 고수준 시맨틱 액션** — search(query), play(title), add_to_cart(item) 등. 정확·저비용이나 사이트별 구현·유지보수 부담, 미지원 사이트 공백
  - **C. 하이브리드** — 고수준 우선, 미지원·실패 시 저수준 폴백
- **예상 평가 축**: 태스크 성공률(NFR-021) / 완료 시간·스텝 수(NFR-030) / 토큰(NFR-008) / 사이트 커버리지·유지보수(NFR-024) / Quick Action 카탈로그·성공 궤적 재사용(TQ-4 전술 6)과 연계

### DP-C8 — 에이전트 루프 구조

- **문제**: 에이전트 루프를 단일 루프로 하는가, 역할 분리 구조로 하는가. 성공률·지연·토큰·테스트 용이성에 동시 영향.
- **후보안(초안)**:
  - **A. 단일 루프(ReAct류)** — 단순·저오버헤드, 긴 태스크에서 컨텍스트 비대·표류 위험
  - **B. Planner–Executor 분리** — 계획은 frontier 모델, 실행은 경량 모델/규칙으로. 토큰 절감·역할별 최적화, 계획-실행 괴리 위험
  - **C. 계층형(오케스트레이터 + 전문 서브루프)** — 복잡 태스크 강함, 오버헤드·복잡도 최대. 50MB TV 환경에서 과설계 위험
- **예상 평가 축**: 성공률(NFR-021) / 완료 시간(NFR-030) / 토큰(NFR-008) / 메모리(NFR-006) / 결정적 재현·테스트(NFR-025)

### DP-C9 — HITL·제어권 모델

- **문제**: 비가역·위험 행동에 대한 사용자 개입을 어떤 정책으로 설계하는가 + 사용자 인터럽트 시 in-flight 행동을 어떻게 중단하는가. 리모컨·음성이라는 TV 입력 특성상 확인 UX 비용이 데스크톱보다 크다.
- **후보안(초안)**:
  - **확인 정책**: A. 위험 등급 기반 선별 확인 / B. 비가역 행동 전 일괄 확인 / C. 가역 행동 선실행 + 취소(undo) 제공
  - **인터럽트 전파**: A. 즉시 강제 중단(hard-kill) — 빠르나 부작용 중간 상태 위험 / B. 협조적 취소(체크포인트 대기) — 일관성 보장하나 중지 지연
- **예상 평가 축**: 안전성(비가역 부작용 방지) / 태스크 완료 시간·UX 마찰(NFR-030) / 개입 시 중지(NFR-034) / 재개(NFR-028) / DP-C6(행동 게이팅)과 연동
- **관련**: 5장 HIL 영역, 6.6 QAS-011

### DP-C10 — 세션 상태·개인화 데이터 보존 모델

- **문제**: 에이전트의 상태(대화 이력·성공 궤적·사용자 선호·사이트 크리덴셜 컨텍스트)를 어디에 얼마나 보존하는가. 개인화·정확도와 프라이버시·규제의 정면 충돌 지점.
- **후보안(초안)**:
  - **A. 휘발(세션 한정)** — 종료 시 전량 폐기. 프라이버시 최상, 매 세션 cold-context로 정확도·효율 손해
  - **B. 온디바이스 영속** — 선호·궤적을 기기 내 저장. 개인화 확보, 저장소(NFR-029)·다중 사용자(가족 TV) 문제
  - **C. 계정 연동 클라우드 동기** — 기기 간 연속성 최상, 프라이버시·규제 부담 최대
- **예상 평가 축**: 정확도·개인화(NFR-020, NFR-021) / PII 보존(NFR-017) / 데이터 주체 권리(NFR-019) / 저장소(NFR-029) / 규제(05-04 compliance matrix) / **TV 특유의 다중 사용자(가족 공용) 문제**
- **관련**: TO-7(트레이스 보존)과 저장소 예산 공유, PRV 영역

## 3. TQ — QA별 설계 전술 탐색 질문

> 원 고민 #4~7. 각각 단일 QA의 "어떻게 달성하나"에 대한 전술 탐색이며, 그 자체로는 아직 DP가 아니다. 전술 풀에서 **구조적 갈림길**(예: 컨텍스트 표현 방식, 상주 모델)이 드러나면 DP로 승격한다. 아래 전술 풀은 브레인스토밍 수준의 후보이며 미검증이다.

### TQ-1 — 메모리 효율화 SW 설계 (원 고민 #4)

- **목표 QA**: Resource Efficiency — NFR-006 (peak ≤ 50MB, 전 라인업 공통) / 관련 TO-3, TO-4
- **전술 후보 풀**:
  1. **상주 최소화 + lazy init** — 컴포넌트를 태스크 시점에 기동, 유휴 시 해제. ↔ NFR-005(슬립 복귀 1초 warm-start)와 정면 긴장
  2. **격리 경계 수 최소화** — 프로세스 분리를 호스트 경계에만 두고 내부는 인프로세스(TO-4 현재 가설과 정합)
  3. **DOM 스냅샷의 부분 직렬화** — 전체 DOM 복사 대신 관심 영역(viewport·인터랙티브 요소)만 추출, 나머지는 참조로 유지
  4. **히스토리·트레이스 디스크 스필** — 대화·관찰 이력을 메모리에 두지 않고 요약본만 상주, 원본은 저장소(NFR-029 예산 내)
  5. **브라우저 엔진 자료구조 재사용** — 엔진이 이미 보유한 접근성 트리 등을 복사 없이 조회(zero-copy 지향)
- **DP 승격 후보 지점**: 전술 1·2가 만나는 **"상주 모델·격리 경계 설계"** — warm-start(NFR-005) ↔ 메모리(NFR-006) ↔ 격리(NFR-009)의 3자 트레이드오프로 구조적 결정임

### TQ-2 — 토큰 효율화 SW 설계 (원 고민 #5)

- **목표 QA**: Resource Efficiency — NFR-008 (세션당 토큰 비용 예산) / 관련 TO-3
- **전술 후보 풀**:
  1. **DOM → 경량 표현 변환** — raw HTML 대신 접근성 트리/인터랙티브 요소 목록 등 압축 표현으로 LLM에 전달 (토큰의 최대 지배 요인)
  2. **컨텍스트 압축·요약(compaction)** — 루프 이력이 길어지면 관찰 결과를 요약으로 치환
  3. **프롬프트 캐싱** — 시스템 프롬프트·도구 정의의 공급사 캐시 활용
  4. **응답 캐싱** — 동일·유사 명령의 계획 결과 재사용
  5. **Quick Action LLM 회피** — 카탈로그 명령은 LLM 호출 자체를 생략 (DP-C2와 직결)
  6. **도구 결과 절단(truncation) 정책** — 관찰 결과의 상한·요약 규칙
- **DP 승격 후보 지점**: 전술 1 — **"페이지 컨텍스트 표현(representation) 방식"**. DOM 표현 선택은 토큰(NFR-008)·정확도(NFR-021)·프라이버시(NFR-018, 전송 범위)를 관통하는 구조적 결정 (TO-2·TO-3·TO-5의 교차점)

### TQ-3 — 수행 속도 개선 SW 설계 (원 고민 #6)

- **목표 QA**: Latency / Performance — NFR-001 (Quick Action ≤1초), NFR-002 (첫 피드백 1초), NFR-005 (슬립 복귀 1초), NFR-030 (카테고리별 완료 시간) / 관련 TO-1
- **전술 후보 풀**:
  1. **Quick Action 우회 경로** — 에이전트 루프·LLM 호출 생략 (DP-C2)
  2. **Pre-warming** — 세션·연결·컨텍스트 사전 준비. ↔ TQ-1 전술 1(상주 최소화)과 긴장
  3. **병렬 도구 호출** — 독립적 관찰·조작의 동시 실행
  4. **스트리밍 + 조기 피드백** — 계획 수립 중 진행 상태 우선 표시(NFR-002 충족, 체감 속도)
  5. **모델 라우팅** — 의도 분류 등 경량 단계에 저지연 모델, 계획에 frontier 모델
  6. **투기적 실행(speculative)** — 고확률 다음 행동의 사전 페치(페이지 프리로드 등)
- **DP 승격 후보 지점**: 전술 2가 TQ-1과 공유하는 **상주 모델** 결정. 그 외는 대체로 병행 적용 가능한 표준 전술(구조적 갈림 아님)

### TQ-4 — 정확성 향상 SW 설계 (원 고민 #7)

- **목표 QA**: Quality / Accuracy — NFR-020 (의도 해석 ≥90%), NFR-021 (태스크 성공률 ≥80%) / 관련 TO-2, TO-5, TO-6
- **전술 후보 풀**:
  1. **멀티모달 grounding** — DOM 텍스트 + 스크린샷 병용으로 요소 식별 정확도 향상. ↔ 토큰 비용(TQ-2)과 긴장
  2. **자기 검증(self-verification) 루프** — 행동 후 결과 관찰·목표 대조·재시도. ↔ 완료 시간(NFR-030)·토큰과 긴장
  3. **HITL 적용 기준** — 불확실·비가역 행동에서 사용자 확인(5장 HIL 영역과 연동)
  4. **typed placeholder 마스킹** — 과마스킹으로 인한 의미 손실 회피 (TO-2 현재 가설)
  5. **사이트별 플레이북/스킬** — 주요 사이트의 검증된 절차를 지식으로 보유. ↔ 유지보수(NFR-024)·확장성과 긴장
  6. **성공 궤적 재사용(trajectory memory)** — 성공한 수행 경로를 기록·재활용. Quick Action 카탈로그의 학습적 확장과 연결
  7. **시뮬레이션 회귀 게이트** — 정확도 퇴행의 조기 검출(NFR-025, TO-6)
- **DP 승격 후보 지점**: 전술 1이 TQ-2와 공유하는 **페이지 컨텍스트 표현 방식**(텍스트 only vs 멀티모달) — 정확도·토큰·지연의 3자 트레이드오프
