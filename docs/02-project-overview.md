# 2. 과제 개요 (Project Overview)

> Tizen TV AI Web Agent (AI Browser) 설계 문서

---

## 2.1 과제 정의

| 항목 | 내용 |
|------|------|
| **과제명** | Tizen TV AI Web Agent (AI Browser) |
| **과제 유형** | 신규 플랫폼 기능 설계 |
| **대상 플랫폼** | Samsung Tizen TV (2억+ 글로벌 설치 기반) |
| **핵심 컨셉** | TV 디바이스에서 동작하는 에이전트 하네스가 AI 모델(클라우드/온디바이스)과 연동하여, 사용자의 음성 명령을 기반으로 웹 브라우저를 자율적으로 제어하는 시스템 |

### 한 줄 정의

> 사용자의 자연어 의도를 이해하고, 웹 브라우저를 자율적으로 조작하며, TV에 최적화된 UI를 동적으로 생성하는 **On-Device 에이전트 하네스 기반의 AI Web Agent**

## 2.3 주요 기능 (Key Features)

### F1. 음성 기반 웹 에이전트 인터랙션

사용자가 리모컨 마이크 또는 상시 대기(always-on) 마이크를 통해 자연어로 웹 작업을 요청하면, AI Web Agent가 이를 해석하고 실행한다.

**주요 능력**:
- 자연어 의도 파싱 및 웹 액션 매핑 (검색, 네비게이션, 클릭, 폼 입력, 스크롤 등)
- 멀티턴 대화를 통한 작업 정제 ("조금 더 싼 것으로 보여줘", "다음 페이지")
- 작업 진행 상태의 시각적/음성 피드백

**예시**: "쿠팡에서 에어컨 필터 검색해서 가격순으로 보여줘"

### F2. Headless Browser 기반 웹 콘텐츠 처리

사용자에게 보이지 않는 백그라운드 브라우저(Headless Browser)에서 웹 페이지를 사전 로드·분석하고, 에이전트가 조작에 필요한 DOM 구조, 인터랙션 포인트, 콘텐츠 시맨틱을 추출한다.

**주요 능력**:
- 웹 페이지 DOM 파싱 및 인터랙터블 요소 식별
- 페이지 콘텐츠 시맨틱 분석 (상품 목록, 폼, 기사 본문 등)
- JavaScript 렌더링 후 완성된 페이지 상태 캡처
- 쿠키/세션 관리를 통한 로그인 상태 유지

### F3. Generative UI 렌더링

AI가 웹 콘텐츠를 분석한 결과를 바탕으로, TV 환경(대화면, 시청 거리, 리모컨 네비게이션)에 최적화된 UI를 동적으로 생성한다. 원본 웹사이트의 레이아웃을 그대로 보여주는 것이 아니라, 사용자의 의도에 맞는 정보를 TV에 적합한 형태로 재구성한다.

**주요 능력**:
- 웹 콘텐츠를 TV 10-feet UI에 맞게 자동 레이아웃 변환
- 사용자 의도에 따른 정보 우선순위 재배치
- 리모컨 D-pad 네비게이션에 최적화된 포커스 구조 자동 생성
- 반응형 카드/리스트/그리드 템플릿 기반 동적 UI 생성

**예시**: 쇼핑 검색 결과를 원본 웹사이트 레이아웃 대신, TV에 최적화된 대형 상품 카드 그리드로 재구성

### F4. 모델 추상화 계층 (Model Abstraction Layer)

에이전트 하네스와 AI 모델 간의 인터페이스를 추상화하여, 클라우드 LLM과 온디바이스 SLM을 동일한 인터페이스로 연동한다. 네트워크 상태, 작업 복잡도, 프라이버시 요구사항에 따라 적절한 모델을 선택하는 라우팅 로직을 포함한다.

**주요 능력**:
- 통합 모델 인터페이스 (요청/응답 포맷 표준화)
- 모델 라우팅: 작업 복잡도·네트워크 상태·프라이버시 기반 자동 선택
- 폴백(Fallback) 메커니즘: 클라우드 불가 시 온디바이스로 자동 전환
- 모델 응답 캐싱 및 결과 후처리 파이프라인

### F5. 자동화 워크플로우 엔진

사용자가 정의한 반복적 웹 작업(구매, 예약, 정보 수집 등)을 에이전트가 자율적으로 수행하는 워크플로우를 지원한다.

**주요 능력**:
- 멀티스텝 웹 작업의 플래닝 및 순차 실행
- 결제/예약 등 민감한 단계에서의 사용자 확인(Human-in-the-Loop)
- 작업 실패 시 자동 재시도 및 대체 경로 탐색
- 작업 완료 결과 요약 및 알림

**예시**: "매주 금요일 CGV에서 이번 주말 상영 영화 목록 정리해줘"

## 2.4 타겟 사용 시나리오

### 시나리오 1: 음성 기반 웹 브라우징

> "네이버에서 이번 주 날씨 알려줘"

1. 사용자가 음성으로 요청
2. 에이전트가 의도 파싱 → Headless Browser로 네이버 날씨 페이지 접근
3. 날씨 정보를 추출하여 Generative UI로 TV에 최적화된 날씨 카드 표시
4. 추가 질문 대기 ("내일은?", "부산 날씨는?")

### 시나리오 2: 자동 구매

> "쿠팡에서 항상 사던 생수 다시 주문해줘"

1. 사용자가 음성으로 구매 요청
2. 에이전트가 과거 구매 이력 또는 저장된 선호도 참조
3. Headless Browser로 쿠팡 접속 → 상품 검색 → 장바구니 추가
4. 결제 직전 단계에서 사용자에게 확인 요청 (Human-in-the-Loop)
5. 사용자 승인 후 결제 완료 → 결과 요약 표시

### 시나리오 3: 예약

> "이번 주 토요일 강남 맛집 예약해줘, 2명"

1. 사용자 요청 파싱 (날짜, 지역, 인원)
2. 에이전트가 맛집 정보 사이트 탐색 → 평점·리뷰 기반 추천 목록 생성
3. Generative UI로 추천 레스토랑 카드 표시
4. 사용자 선택 후 예약 사이트 접속 → 예약 폼 자동 입력
5. 최종 확인 후 예약 완료

### 시나리오 4: TV 브라우저 음성 제어

> "스크롤 내려줘", "저 파란 버튼 클릭해", "뒤로 가줘"

1. 현재 표시 중인 웹 페이지에 대해 직접적인 브라우저 조작 명령
2. 에이전트가 명령을 브라우저 액션으로 즉시 변환
3. 포커스 이동, 클릭, 스크롤, 뒤로가기 등 실시간 실행
4. 복합 명령도 지원 ("아래로 좀 내려서 '자세히 보기' 클릭해줘")

### 시나리오 5: Headless + Generative UI 기반 콘텐츠 재가공

> 기존 웹앱의 품질이 TV에 부적합한 경우

1. 웹앱 로딩 시 Headless Browser에서 페이지를 사전 분석
2. 레이아웃, 폰트, 인터랙션 요소가 TV 환경에 부적합함을 감지
3. AI가 콘텐츠를 추출하고 TV 최적화된 Generative UI로 재구성
4. 사용자에게는 원본 대신 최적화된 UI를 표시 (원본 전환 옵션 제공)

### 핵심 컴포넌트 요약

| 컴포넌트 | 위치 | 역할 |
|----------|------|------|
| **Voice Input** | TV 디바이스 | 사용자 음성 캡처 및 STT 변환 |
| **Agent Harness** | TV 디바이스 (On-Device) | 에이전트 코어 — 의도 파싱, 액션 실행, 워크플로우 오케스트레이션 |
| **Model Abstraction Layer** | TV 디바이스 | 클라우드/온디바이스 모델 추상화, 라우팅, 폴백 |

| **Generative UI** | TV 디바이스 | 웹 콘텐츠를 TV 최적화된 UI로 동적 생성 |
| **Cloud LLM API** | 클라우드 | 복잡한 추론, 자연어 이해, 플래닝 처리 |
| **On-Device SLM** | TV 디바이스 (Optional) | 간단한 의도 분류, 캐시된 응답, 오프라인 기본 동작 |

## 2.6 주요 기술 스택

| 영역 | 기술 | 선정 근거 |
|------|------|-----------|
| **에이전트 런타임** | Chromium Embedded + Custom Agent Runtime | Tizen 기본 웹 엔진 활용, 브라우저 제어 통합 |
| **Headless Browser** | Chromium CDP (Chrome DevTools Protocol) | DOM 접근, 네트워크 제어, JavaScript 실행에 표준화된 프로토콜 |
| **Generative UI** | Web Components + Dynamic Template Engine | TV 웹 런타임과 호환, 동적 UI 생성에 적합 |
| **음성 처리** | Tizen Platform STT API + 커스텀 NLU 파이프라인 | 기존 TV 음성 인프라 활용, 에이전트 특화 NLU 추가 |
| **모델 연동** | REST/gRPC 기반 Model API Gateway | 클라우드·온디바이스 모델 통합 인터페이스 |
| **워크플로우** | 경량 상태 머신 (State Machine) 엔진 | TV 리소스 제약 하에서 멀티스텝 작업 관리 |
| **데이터 저장** | 로컬 IndexedDB + Encrypted Storage | 사용자 선호도, 세션, 캐시 데이터의 안전한 로컬 저장 |

## 2.7 주요 제약사항 및 고려사항

### 기술적 제약사항

| 제약사항 | 영향 | 대응 전략 |
|----------|------|-----------|
| **TV SoC 리소스 한계** | RAM 1~2GB, CPU 제한적 | 에이전트 하네스 경량화, 메모리 풀 관리, 지연 로딩 |
| **네트워크 의존성** | 클라우드 LLM 사용 시 네트워크 필수 | 온디바이스 SLM 폴백, 응답 캐싱, 오프라인 기본 모드 |
| **브라우저 엔진 제약** | Tizen Chromium 버전 제한 | 지원 API 범위 내 설계, 폴리필 적용 |
| **입력 장치 제한** | 리모컨 D-pad + 음성만 가능 | 음성 우선 UX, D-pad 폴백 네비게이션 설계 |

### 보안 및 프라이버시

| 고려사항 | 대응 방안 |
|----------|-----------|
| 사용자 웹 브라우징 데이터 | 하네스 내 로컬 처리 우선, 클라우드 전송 시 익명화 및 최소 데이터 원칙 |
| 결제/로그인 자격 증명 | 에이전트가 직접 저장하지 않음, OS 레벨 Credential Manager 연동 |
| AI 모델에 전달되는 컨텍스트 | 민감 정보 필터링 파이프라인 적용, PII 마스킹 |
| 자동 구매/예약 | Human-in-the-Loop 필수 적용, 금액 한도 설정 |
